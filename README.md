# F2F Bank — E2E тесты (Playwright)

Автоматизированные end-to-end тесты для приложения F2F Bank,
реализованные на Playwright + TypeScript.

## Стек приложения
- **database** — PostgreSQL
- **app** — бэкенд на Python/FastAPI
- **client** — фронтенд на Vue.js
- **web-proxy** — Nginx (раздаёт приложение на `http://localhost`)

## Предварительные требования

| Инструмент | Версия | Проверка |
|---|---|---|
| Docker Desktop | любая актуальная | `docker --version` |
| Docker Compose | v2+ | `docker compose version` |
| Node.js | 18+ | `node --version` |
| Git | любая | `git --version` |

## Как запустить приложение

```powershell
git clone <ссылка-на-этот-репозиторий>
cd f2f-bank-tests
docker compose up -d --build
```

Первая сборка занимает 2–5 минут. После запуска приложение доступно на
`http://localhost`.

Проверить, что всё поднялось:
```powershell
docker compose ps
```

Остановить:
```powershell
docker compose down
```

Полный сброс вместе с данными базы:
```powershell
docker compose down -v
```

## Как запустить тесты

Установить зависимости (если ещё не установлены):
```powershell
npm install
npx playwright install
```

Запустить весь набор тестов (во всех браузерах: Chromium, Firefox, WebKit):
```powershell
npx playwright test
```

Запустить только в Chromium:
```powershell
npx playwright test --project=chromium
```

Запустить конкретный файл:
```powershell
npx playwright test transfer.spec.ts
```

Запуск в UI-режиме (визуальный, удобен для отладки):
```powershell
npx playwright test --ui
```

Посмотреть HTML-отчёт после прогона:
```powershell
npx playwright show-report
```

> Приложение должно быть запущено (`docker compose up -d`) перед
> запуском тестов — конфиг Playwright обращается к `http://localhost`.

## Описание тест-кейсов и приоритетов

| Файл | Сценарий | Приоритет |
|---|---|---|
| `registration.spec.ts` | Успешная регистрация нового пользователя | Критический |
| `login.spec.ts` | Успешный вход с валидными данными | Критический |
| `login.spec.ts` | Вход с неверным паролем отклонён | Высокий |
| `transfer.spec.ts` | Успешный перевод при достаточном балансе | Критический |
| `transfer.spec.ts` | Перевод при недостаточном балансе отклонён | Критический |
| `transactions.spec.ts` | Пополнение баланса появляется в истории | Высокий |
| `transactions.spec.ts` | Перевод появляется в истории транзакций | Высокий |
| `transactions.spec.ts` | У нового пользователя нет транзакций | Средний |
| `profile.spec.ts` | Профиль отображает корректные данные пользователя | Низкий |
| `validation.spec.ts` | Перевод с номером телефона без "+" отклонён | Высокий |
| `validation.spec.ts` | Перевод с суммой 0 отклонён | Средний |
| `validation.spec.ts` | Перевод с отрицательной суммой отклонён | Средний |
| `validation.spec.ts` | Регистрация с уже существующим email отклонена | Высокий |
| `validation.spec.ts` | Некорректный формат email отклонён | Средний |

**Логика расстановки приоритетов:**
- **Критический** — базовые сценарии, без которых приложение
  неработоспособно (регистрация, вход, денежный перевод и его защита
  от ухода в минус по балансу).
- **Высокий** — защита от некорректных данных при денежных операциях
  и дублирующихся аккаунтов, отображение истории операций.
- **Средний** — дополнительная валидация полей форм.
- **Низкий** — некритичный для основного flow функционал (просмотр
  профиля).

## Технические детали реализации тестов
- Общие для тестов действия (регистрация + вход, пополнение баланса)
  вынесены в `tests/helpers.ts`, чтобы не дублировать код.
- Все клики, инициирующие сетевые запросы (Register/Login/Send/Add),
  сопровождаются явным ожиданием ответа сервера
  (`page.waitForResponse`) — это устраняет гонки между действием
  в браузере и его обработкой на бэкенде.
- Локаторы построены на семантических атрибутах (`getByRole`,
  `getByPlaceholder`, `getByText`) — устойчивы к изменениям вёрстки,
  не зависящим от точной структуры DOM.
- Найденные в ходе тестирования особенности приложения задокументированы
  в [`NOTES.md`](./NOTES.md).
